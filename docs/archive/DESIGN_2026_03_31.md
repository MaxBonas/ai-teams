## Contexto y motivación

Durante una sesión de prueba ("Prueba AI Teams") se analizó una corrida real de recuperación de contexto.
El flujo fue: usuario preguntó "¿de qué iba el proyecto?" → el sistema corrió un workflow completo de
326 segundos → resultado rechazado (P30/100) → Lead Close fabricó hechos que el Researcher había
desmentido explícitamente.

Se identificaron 7 puntos débiles estructurales:

| # | Problema | Severidad | Estado |
|---|----------|-----------|--------|
| 1 | Lead Close fabrica hechos no respaldados por Researcher | CRÍTICO | ✅ Mitigado (prompt) |
| 2 | Researcher no lee archivos del proyecto, solo historial de chat | Alto | ✅ Mitigado (prompt) |
| 3 | Workflow completo para una pregunta de orientación | Alto | ✅ Mitigado (bypass gate) |
| 4 | Lead Intake no hace preguntas aclaratorias cuando el objetivo es ambiguo | Medio | ✅ Mitigado (prompt) |
| 5 | Quality gate calibrado para build runs; rechaza runs de contexto válidos | Medio | ✅ Mitigado (bypass) |
| 6 | Output del inspector truncado sin indicación visual | Bajo | ✅ Fix (1500→3000 chars) |
| 7 | Lead Close puede ignorar condiciones del QA | Medio | ✅ Mitigado (prompt) |

Los ítems 1-7 fueron mitigados con cambios en prompts y lógica de scoring (commits 2026-03-31).
Sin embargo, la causa raíz estructural no está resuelta: el Team Lead (modelo caro) hace trabajo
de bajo nivel que debería delegar.

---

## Problema estructural raíz

El Team Lead actualmente hace **dos trabajos incompatibles**:

1. **Inteligencia cara**: decidir, descomponer, sintetizar, coordinar
2. **Trabajo de bajo nivel**: recuperar archivos, leer historial, explorar contexto, hacer grep

Consecuencias:
- Tokens caros desperdiciados en lectura de contexto raw
- Lead recibe contexto sin filtrar → riesgo de alucinación por overflow / truncado
- Scoring de "productividad" no distingue tipos de trabajo → rechaza runs válidos
- No hay separación de preocupaciones entre capas

## Arquitectura propuesta: Three-Tier Delegation

```
┌──────────────────────────────────────────────────────────────┐
│  TIER 1 — COMMAND (modelo caro, pocas llamadas)              │
│  Team Lead: solo recibe briefings, decide, coordina           │
│  Modelos: claude-opus / gpt-4o (senior_cloud)                │
│  Input: briefings compactos ≤ 800 chars por fuente           │
│  Output: decisiones, planes, síntesis ejecutiva              │
└─────────────────────┬────────────────────────────────────────┘
                      │ delega vía plan estructurado
                      ▼
┌──────────────────────────────────────────────────────────────┐
│  TIER 2 — SPECIALIST (modelo medio)                          │
│  Engineer / Reviewer / QA: implementan, revisan, verifican   │
│  Modelos: claude-sonnet / gpt-4o-mini (advanced_api)         │
│  Output: código, reportes de revisión, resultados de QA      │
└─────────────────────┬────────────────────────────────────────┘
                      │ delega investigación
                      ▼
┌──────────────────────────────────────────────────────────────┐
│  TIER 3 — SCOUT (modelo barato / gratuito)                   │
│  Lee archivos, historial de sesiones, git status, grep        │
│  Modelos: claude-haiku / gemini-flash-lite / local           │
│  Output siempre compacto: ≤ 300 chars por fuente             │
│  Costo objetivo: ~$0.001 por briefing completo               │
└──────────────────────────────────────────────────────────────┘
```

### Nuevo rol: SCOUT

```python
class Role(str, Enum):
    TEAM_LEAD  = "team_lead"
    RESEARCHER = "researcher"
    ENGINEER   = "engineer"
    REVIEWER   = "reviewer"
    QA         = "qa"
    SCOUT      = "scout"   # NUEVO — solo en Tier 3
```

Características del rol SCOUT:
- Router siempre asigna `budget_api` o `local` (nunca senior_cloud ni advanced_api)
- Output máximo: 800 tokens por tarea
- Sin quality gates (no genera código, no necesita revisión)
- Sin evidence gates (su valor es informativo, no artefactual)
- Siempre corre en paralelo con otros scouts de la misma fase
- No puede delegar a otros agentes

### Tipos de scout task

| Tipo | Descripción | Output esperado |
|------|-------------|----------------|
| `scout_project_state` | git status + últimos 5 archivos modificados + estructura clave | ~300 chars |
| `scout_session_history` | Decisiones relevantes de las últimas 2-3 sesiones | ~400 chars |
| `scout_code_section` | Leer sección específica de código (pedido por Lead/Researcher) | ~500 chars |
| `scout_search` | grep/find en el proyecto con query específica | ~300 chars |

## Patrón A: Pre-flight scouts (MVP — implementar ahora)

Antes de que el Team Lead ejecute `lead_intake`, siempre se lanzan scouts en paralelo:

```
Usuario envía mensaje
        │
        ▼
┌─────────────────────────────────────────┐
│  PRE-FLIGHT PHASE (paralelo)            │
│  scout_project_state  → briefing_A      │
│  scout_session_history → briefing_B     │
└────────────────┬────────────────────────┘
                 │ ambos completan en ~5-15s
                 ▼
┌─────────────────────────────────────────┐
│  lead_intake                            │
│  Recibe: briefing_A + briefing_B        │
│  (no lee nada directamente)             │
└─────────────────────────────────────────┘
```

**Ventajas**:
- Lead siempre parte con contexto real del proyecto
- Scouts corren baratos/gratis; no consumen budget caro
- Latencia: scouts corren en paralelo ~10-20s; actualmente el Lead tarda 60-90s haciendo lo mismo
- Elimina el riesgo de alucinación por contexto incompleto

**Implementación**:
1. Antes de crear `lead_intake`, insertar `CHAT-xxx::scout_project_state` y `CHAT-xxx::scout_session_history`
2. `lead_intake` depende de ambas tareas scout
3. El contexto de dependencias lleva los briefings al Lead (ya existe `_build_dependency_output_context`)

## Reforma del scoring (Productivity Score)

### El problema

El P-score actual está calibrado para **runs de build**. No distingue tipos de trabajo:

| Tipo de run | P-score actual | ¿Correcto? |
|---|---|---|
| Context recovery → respuesta correcta | P=0–30, REJECTED | ❌ |
| Scout lee archivos + Lead sintetiza | P=0, REJECTED | ❌ |
| Engineer escribe 3 líneas de código | P=45, OK | ~ (sobrevalora artefactos) |
| Lead fabrica hechos con elegancia | P=60, OK | ❌ (output falso aceptado) |

### Scoring por tipo de run

```
run_type = detect_run_type(message, phases, artifacts)
```

| `run_type` | Métrica primaria | Threshold |
|---|---|---|
| `context_recovery` | reasoning_score ≥ 50 | R≥50 → accepted |
| `planning` | reasoning_score ≥ 60 + phase_coverage ≥ 80% | — |
| `build` | productivity_score (actual) | P≥35 → accepted |
| `scout_run` | scout_completeness (% scouts completados) | ≥80% → accepted |
| `mixed` | max(P, R) ≥ 40 | — |

**Implementación**: `_detect_run_type(payload, phase_task_ids, artifacts)` → enum
Usado en el bloque de evaluación de calidad para seleccionar el threshold correcto.

### Coherencia de políticas

Lo que se quiere que sea como AI Teams de cloud pero multimodelo:
- ✅ Team Lead siempre usa modelo avanzado
- ✅ Especialistas usan modelo adecuado a la tarea
- ✅ Scouts usan modelos baratos/gratis
- ✅ Cada tipo de run tiene su propia definición de éxito
- ✅ El valor de una corrida = trabajo realizado correctamente, no solo artefactos producidos
- ❌ (actual) Un run de orientación válido se rechaza por no tener código
- ❌ (actual) El Lead consume tokens caros leyendo archivos directamente

## Patrón D: Pausa conversacional del Lead (E7-D)

### Motivación

El Lead a veces no tiene suficiente información para planificar. Hoy la única salida es
inventar fases o hacer suposiciones. Con este patrón, el Lead puede emitir una pregunta
aclaratoria estructurada y el flujo se pausa hasta que el usuario responda.

### Mecanismo

```
lead_intake output:
  "Antes de planificar necesito saber: ¿el endpoint debe ser REST o GraphQL?
   [CLARIFY: "¿el endpoint debe ser REST o GraphQL?"]"

→ api/main.py detecta [CLARIFY: "..."]
→ devuelve {"status": "waiting_user", "question": "..."}
→ guarda estado en runtime/pending_clarification_{chat_id}.json
→ frontend muestra pregunta y campo de respuesta

Usuario responde → POST /api/chat/clarify {chat_id, clarification}
→ estado cargado, respuesta inyectada en contexto del Lead como:
  "[Respuesta del usuario a tu pregunta: 'REST']"
→ lead_intake re-ejecuta (o continúa en siguiente sub-iteración)
→ run sigue normalmente
```

### Estados del run (MVP)

```
running → waiting_user → running → completed
                ↑
          usuario responde
```

### Contrato de API (MVP)

**Respuesta pausada** (nueva):
```json
{"status": "waiting_user", "question": "¿REST o GraphQL?", "chat_id": "CHAT-xxx"}
```

**Endpoint de reanudación** (nuevo):
```
POST /api/chat/clarify
{"chat_id": "CHAT-xxx", "clarification": "REST, con autenticación JWT"}
```

**Respuesta normal** (sin cambio):
```json
{"status": "completed", "result": "...", ...}
```

### Archivos afectados (E7-D MVP)

| Archivo | Cambio |
|---------|--------|
| `api/main.py` | Detectar `[CLARIFY: "..."]` en output de lead_intake; guardar estado parcial; endpoint `/api/chat/clarify` |
| `ide-frontend/src/` | Si `status == "waiting_user"`, mostrar pregunta + input de respuesta |
| `runtime/` | Archivo `pending_clarification_{chat_id}.json` con estado parcial |

### Invariantes de diseño (post E7-D4/D5 — mid-run completo)

- **E7-D1/D2/D3 (MVP)**: `[CLARIFY]` en lead_intake → pausa → respuesta → re-run desde lead_intake. Implementado 2026-03-31.
- **E7-D4/D5 (Fase 2)**: `[CLARIFY]` en CUALQUIER fase (excepto lead_intake, lead_close, scouts) → `TaskState.WAITING_USER` en orquestador → `_refresh_readiness` congela tareas downstream → `api/main.py` detecta post-PASO-3 → guarda `type=mid_run` en `pending_clarification_{root}.json` → `_build_resume_stream` reanuda con SSE streaming desde el punto de pausa.
- `WAITING_USER` se serializa como `"waiting_user"` en JSON y survives reloads del taskboard.
- El frontend usa el mismo `sendClarification()` para ambos paths — la distinción es interna (`type` en el pending state).
- Si el usuario no responde (timeout de sesión), el estado queda guardado indefinidamente — el usuario puede responder más tarde.
- Máximo una pausa activa por run (la primera tarea WAITING_USER bloquea las demás).

### Riesgos

| Riesgo | Mitigación |
|--------|-----------|
| Lead emite `[CLARIFY]` en runs donde tiene suficiente info | Prompt instruction: usar `[CLARIFY]` solo si la ambigüedad bloqueará la planificación |
| Estado pendiente acumulado en `runtime/` | TTL de 24h; limpiar estados viejos en startup |
| Frontend no maneja `waiting_user` y muestra error | Gestión explícita del estado en UI antes de implementar E7-D3 |

## EPIC-8: Lead Control Protocol (LCP)

**Fecha**: 2026-03-31
**Estado**: Diseñado. E8-A (MVP) pendiente implementación.

### Motivación

El flujo actual es rígido: `scouts → lead_intake → fases → lead_close`. El Lead ejecuta ese
pipeline independientemente de lo que necesite realmente el usuario. Consecuencias:

- Una pregunta de orientación lanza researcher + qa + close (innecesario y caro)
- Una tarea crítica pasa por los mismos gates que una trivial
- Si el Lead "sabe" que necesita una fase extra, no puede añadirla
- Si el Lead ve que el plan es excesivo, no puede simplificarlo

El LCP da al Lead **control total sobre el flujo** a través de directivas estructuradas que
el orquestador interpreta y ejecuta. El Lead es el único árbitro.

### Protocolo de directivas

```
[DIRECTIVE: payload]
```

El orquestador escanea el output de `lead_intake` tras PASO 1 (antes de construir fases).
Las directivas no aparecen en la respuesta al usuario — se extraen y se eliminan.

### Directivas y su semántica exacta

#### `[DIRECT_ANSWER]`
**Cuándo**: La respuesta está completa en los briefings scout o en el conocimiento del Lead.
No se necesita ningún agente.
```
Efecto: phase_task_ids = {lead_intake: ...}  ← sin fases dinámicas ni lead_close
        response = output del Lead (sin la directiva)
        state = "completed"
```
**Caso de uso**: "¿qué hace esta función?" / "explícame el error que me diste" / "¿de qué iba?"

#### `[REJECT: "razón"]`
**Cuándo**: La petición está fuera de scope, es imposible, o violaría guardrails.
```
Efecto: igual que DIRECT_ANSWER pero state = "rejected"
        response = explicación clara de por qué no puede ejecutarse
```
**Caso de uso**: "hackea esta web" / "borra toda la DB en producción sin backup" / "no entiendo qué quieres"

#### `[ABORT_PHASES: "razón"]`
**Cuándo**: La respuesta ya es completa con lo que saben los scouts y el Lead.
No vale la pena lanzar más agentes.
```
Efecto: igual que DIRECT_ANSWER — el Lead ya tiene todo lo necesario
        Diferencia semántica: ABORT_PHASES implica "podría haber fases, pero no hacen falta"
```
**Caso de uso**: "¿hay errores en el proyecto?" → scouts ya lo ven → Lead responde directamente

#### `[CLARIFY: "pregunta"]` *(ya implementado — E7-D)*
**Cuándo**: Hay ambigüedad que bloquearía la planificación.
```
Efecto: state = "waiting_user", pregunta guardada, run pausado
```

#### `[ESCALATE: complexity=X criticality=X]`
**Cuándo**: El Lead estima que la tarea es más compleja/crítica de lo que el usuario indicó.
`X` puede ser: `low | medium | high | critical` (criticality) o `low | medium | high | very_high` (complexity).
```
Efecto: complexity y criticality de TODAS las WorkTask dinámicas actualizadas
        round_budget ajustado: high → +50%, very_high → +100%
        gates más estrictos (criticality=critical requiere doble aprobación en prod)
```
**Caso de uso**: "añade auth JWT" → Lead ve que hay sesiones, RBAC, refresh tokens → [ESCALATE: complexity=high]

#### `[SKIP: "phase_id1 phase_id2"]`
**Cuándo**: Una o más fases del plan son innecesarias para este caso específico.
```
Efecto: esas fases se eliminan del plan antes de crear WorkTasks
        las dependencias se reajustan automáticamente
```
**Caso de uso**: cambio de documentación → [SKIP: "review qa"] / consulta de análisis → [SKIP: "build"]

#### `[ADD_PHASE: role "objetivo detallado"]`
**Cuándo**: El Lead ve que falta una fase que el WORKFLOW_PLAN no incluyó.
`role` = RESEARCHER | ENGINEER | REVIEWER | QA
```
Efecto: nueva WorkTask insertada al final del plan dinámico (antes de lead_close)
```
**Caso de uso**: "migra a PostgreSQL" → Lead ve que falta fase de backup → [ADD_PHASE: ENGINEER "Exportar datos actuales"]

#### `[EXTEND_BUDGET: +N]`
**Cuándo**: El Lead estima que la tarea necesita más rondas de las asignadas.
```
Efecto: round_budget += N (máximo: round_budget * 3 para evitar runs infinitos)
        emite evento `lcp_budget_extended`
```
**Caso de uso**: tarea de refactoring masiva → [EXTEND_BUDGET: +15]

#### `[FORCE_GATE: "phase_id"]` *(E8-B)*
**Cuándo**: Lead decide que una fase necesita re-revisión.
```
Efecto: añade gate adicional a la fase especificada (si ya completó)
```

#### `[REPLAN]` *(E8-B)*
**Cuándo**: El plan inicial era incorrecto. Lead emite nuevo WORKFLOW_PLAN completo.
```
Efecto: reemplaza el plan dinámico si ninguna fase ha empezado
        si ya empezaron fases: solo añade fases al final (safe subset)
```

### Flujo de procesamiento

```
PASO 1: scouts + lead_intake
         ↓
_extract_lcp_directives(lead_output)
         ↓
¿REJECT?      → return rejected response
¿ABORT/DIRECT → return completed response (lead_output sin directivas)
¿CLARIFY?     → save state, return waiting_user
         ↓
¿ESCALATE?    → update complexity/criticality/round_budget
¿SKIP?        → filtrar fases del WORKFLOW_PLAN
¿ADD_PHASE?   → añadir fase al plan
¿EXTEND?      → round_budget += N
         ↓
PASO 2: construir fases dinámicas con plan ajustado
PASO 3: ejecutar fases + lead_close
```

### Instrucción al Lead (profiles.py — extracto)

```
DIRECTIVAS DE FLUJO (emitir al final de lead_intake):

[DIRECT_ANSWER]    — si la respuesta no requiere agentes (orientacion, explicacion, contexto)
[REJECT: "razon"]  — si la peticion esta fuera de scope o es imposible
[ABORT_PHASES: "razon"] — si los scouts ya dan suficiente informacion para responder
[CLARIFY: "pregunta"] — si la ambiguedad bloqueara el plan (max 1 pregunta)
[ESCALATE: complexity=high criticality=critical] — si la tarea es mas critica de lo indicado
[SKIP: "review qa"] — para eliminar fases innecesarias (separar con espacio)
[ADD_PHASE: ENGINEER "objetivo"] — para añadir una fase que el plan omitio
[EXTEND_BUDGET: +10] — si necesitas mas rondas

Reglas:
- Solo emite directivas que sean necesarias. Si el flujo estandar es correcto, no emitas ninguna.
- [DIRECT_ANSWER] tiene precedencia sobre WORKFLOW_PLAN — no planifiques fases si lo usas.
- Las directivas se eliminan de la respuesta al usuario automaticamente.
- Puedes combinar: [ESCALATE: complexity=high] [SKIP: "review"]
```
