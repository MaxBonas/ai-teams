# Diseño: Visibilidad Per-Agent y Workflow Dinámico

**Fecha**: 2026-03-28
**Estado**: ✅ Implementado — ver `docs/TASKS_2026_03_28.md` para detalles de implementación
**Relacionado**: `docs/AUDIT_2026_03_27.md` (issues M3, M5), `docs/TASKS_2026_03_28.md`

---

## Contexto y motivación

El sistema actual tiene dos limitaciones estructurales que lo separan de una experiencia de "equipo real":

1. **El usuario es ciego al proceso** — ve una barra de progreso, pero no sabe qué está pensando cada agente ni por qué tomó una decisión.
2. **El workflow es una receta fija** — las fases son siempre las mismas independientemente de la complejidad o naturaleza del pedido. Un fix de un bug recorre las mismas 8 fases que un feature complejo.

Estos dos problemas se abordan en este documento.

---

## Parte A: Visibilidad per-agent (Agent Lanes)

### Objetivo

El usuario ve en tiempo real qué está haciendo cada agente: qué rol tiene, en qué fase está, qué está "pensando" y qué produce. Similar al panel de reasoning de Claude en modo extended thinking.

### Estado actual del pipeline de streaming

```
adapter.invoke_stream(on_chunk=fn)
  └─ router.route_and_invoke(on_chunk=fn)          [router.py:450-456]
       └─ orchestrator._on_chunk(task_id, chunk)   [orchestrator.py:945-949]
            └─ api._on_chunk(task_id, chunk)        [api/main.py:1874-1876]
                 └─ queue.put(("token_chunk", {...}))
                      └─ SSE stream → frontend
```

El payload actual de `token_chunk`:
```json
{"task_id": "CHAT-xyz::build", "chunk": "fragmento de texto"}
```

No hay metadata de rol, agente, ni tipo de chunk (razonamiento vs output).

### Diseño propuesto

#### A1. Nuevos tipos de evento SSE

| Evento | Cuándo | Payload |
|--------|--------|---------|
| `agent_started` | Agente toma ownership de una tarea | `{task_id, agent_id, role, phase, title}` |
| `agent_chunk` | Token del agente (reemplaza token_chunk) | `{task_id, agent_id, role, phase, chunk, chunk_type}` |
| `agent_completed` | Tarea completada | `{task_id, agent_id, role, phase, preview, duration_ms}` |
| `agent_failed` | Tarea fallida | `{task_id, agent_id, role, phase, error}` |

`chunk_type` puede ser:
- `"output"` — texto que forma parte de la respuesta final
- `"thinking"` — razonamiento intermedio (no entra en el output)
- `"tool_call"` — llamada a herramienta en curso

`token_chunk` se mantiene por backward compatibility (mismo contenido que `agent_chunk` con chunk_type=output).

#### A4. Soporte de thinking chunks (Claude extended thinking)

Los adapters de Claude API ya soportan `thinking` blocks. En `aiteam/adapters/api.py`, cuando se recibe un bloque de tipo `thinking`, emitirlo como `chunk_type: "thinking"`:

```python
# En el parser de streaming de la API de Anthropic
if block.type == "thinking":
    on_chunk_with_type(block.thinking, "thinking")
elif block.type == "text":
    on_chunk_with_type(block.text, "output")
```

Esto requiere que el callback acepte `(chunk, chunk_type)` en lugar de solo `(chunk)`. Usar un wrapper para backward compatibility.

**Handler de SSE nuevos eventos:**
```typescript
case 'agent_started':
  setAgentLanes(prev => new Map(prev).set(data.task_id, {
    taskId: data.task_id,
    agentId: data.agent_id,
    role: data.role,
    phase: data.phase,
    title: data.title,
    status: 'thinking',
    outputText: '',
    thinkingText: '',
    preview: '',
    startedAt: Date.now(),
  }));
  break;

case 'agent_chunk':
  setAgentLanes(prev => {
    const lane = prev.get(data.task_id);
    if (!lane) return prev;
    const updated = { ...lane, status: 'outputting' as const };
    if (data.chunk_type === 'thinking') {
      updated.thinkingText = lane.thinkingText + data.chunk;
    } else {
      updated.outputText = lane.outputText + data.chunk;
    }
    return new Map(prev).set(data.task_id, updated);
  });
  break;

case 'agent_completed':
  setAgentLanes(prev => {
    const lane = prev.get(data.task_id);
    if (!lane) return prev;
    return new Map(prev).set(data.task_id, {
      ...lane,
      status: 'completed',
      preview: data.preview,
      durationMs: data.duration_ms,
    });
  });
  break;
```

## Parte B: Workflow dinámico (Dynamic Phases)

### Objetivo

El Team Lead analiza cada pedido y propone las fases apropiadas. Un fix rápido puede tener 3 fases; un feature nuevo puede necesitar 10. Las fases y sus roles emergen del razonamiento del Team Lead, no de un enum hardcodeado.

### Estado actual de las fases

En `api/main.py:1963-2014`, hay dos bloques if/else con listas hardcodeadas:
- **classic**: `lead_intake → discovery → build → review → qa → lead_close`
- **sprint5**: `lead_intake → plan_research → plan_engineering → plan_risks → build → review → qa → lead_close`

La función `_delegation_brief(phase)` tiene un dict literal con los objetivos por fase.

El Team Lead no tiene forma de influir en el plan.

### Diseño propuesto

#### B1. Flujo de dos pasos

```
Paso 1: Crear y ejecutar SOLO lead_intake
         ↓
         Lead analiza el pedido y emite un WORKFLOW_PLAN
         ↓
Paso 2: Parsear WORKFLOW_PLAN → crear WorkTasks dinámicamente
         ↓
         Ejecutar las fases generadas
```

#### B2. Formato de WORKFLOW_PLAN en el output del Lead

Agregar al prompt de `lead_intake` la instrucción de generar un plan:

```
Al final de tu análisis, incluye un bloque WORKFLOW_PLAN con las fases necesarias
para este pedido específico. Usa solo los roles disponibles: RESEARCHER, ENGINEER,
REVIEWER, QA. Sé preciso: no incluyas fases innecesarias.

Formato obligatorio:
[WORKFLOW_PLAN]
- phase_id: research
  role: RESEARCHER
  objective: Investigar restricciones y APIs disponibles
  depends_on: []
- phase_id: implement
  role: ENGINEER
  objective: Implementar el endpoint con validación de inputs
  depends_on: [research]
- phase_id: qa
  role: QA
  objective: Verificar flujo end-to-end con casos edge
  depends_on: [implement]
[/WORKFLOW_PLAN]
```

Restricciones del plan:
- `lead_intake` y `lead_close` son siempre insertados automáticamente (no los incluye el Lead)
- Mínimo 1 fase, máximo 10
- `depends_on` debe referenciar solo phase_ids definidos en el mismo plan
- No puede haber ciclos en las dependencias

#### B3. Parser del WORKFLOW_PLAN

Nueva función en `aiteam/workflow_planner.py` (archivo nuevo):

```python
from dataclasses import dataclass
from typing import List
import re

@dataclass
class PhaseSpec:
    phase_id: str
    role: str           # "RESEARCHER" | "ENGINEER" | "REVIEWER" | "QA"
    objective: str
    depends_on: List[str]

def parse_workflow_plan(lead_output: str) -> List[PhaseSpec] | None:
    """
    Extrae el WORKFLOW_PLAN del output del Team Lead.
    Retorna None si no hay plan válido (usar fases por defecto).
    """
    match = re.search(
        r'\[WORKFLOW_PLAN\](.*?)\[/WORKFLOW_PLAN\]',
        lead_output,
        re.DOTALL
    )
    if not match:
        return None

    block = match.group(1).strip()
    phases = _parse_phase_blocks(block)

    if not phases:
        return None

    if not _validate_phases(phases):
        return None  # fallback a default

    return phases

def _validate_phases(phases: List[PhaseSpec]) -> bool:
    ids = {p.phase_id for p in phases}
    for p in phases:
        if p.role not in {"RESEARCHER", "ENGINEER", "REVIEWER", "QA"}:
            return False
        for dep in p.depends_on:
            if dep not in ids:
                return False
    return _no_cycles(phases)
```

#### B4. Registro de roles por defecto

Nueva función/dict en `aiteam/workflow_planner.py`:

ROLE_ALIASES: dict[str, str] = {
    # RESEARCHER
    "research": "RESEARCHER", "discovery": "RESEARCHER", "analysis": "RESEARCHER",
    "investigate": "RESEARCHER", "explore": "RESEARCHER",
    # ENGINEER
    "build": "ENGINEER", "implement": "ENGINEER", "develop": "ENGINEER",
    "code": "ENGINEER", "fix": "ENGINEER", "refactor": "ENGINEER",
    # REVIEWER
    "review": "REVIEWER", "security": "REVIEWER", "audit": "REVIEWER",
    "validate_design": "REVIEWER",
    # QA
    "qa": "QA", "test": "QA", "verify": "QA", "acceptance": "QA",
}

def infer_role_from_phase_id(phase_id: str) -> str:
    """Infiere rol por coincidencia de alias. Default: ENGINEER."""
    lower = phase_id.lower()
    for keyword, role in ROLE_ALIASES.items():
        if keyword in lower:
            return role
    return "ENGINEER"

## Dependencias entre features

```
Feature A (visibilidad) depende de:
  - Orquestador emita eventos de ciclo de vida → A2
  - API propague los eventos → A3
  - Frontend los consuma → A5

Feature B (workflow dinámico) depende de:
  - Nuevo aiteam/workflow_planner.py → B3, B4, B6
  - Cambio en api/main.py (flujo de dos pasos) → B5
  - Prompt del lead_intake actualizado → B2
  - Frontend manejo del estado "planning" → B7
```

Las dos features son **independientes**: pueden implementarse en cualquier orden. Se recomienda empezar por **B (workflow dinámico)** porque es el cambio más estructural y hace que A sea más valioso (tiene sentido ver el razonamiento de fases no triviales).

## Riesgos y mitigaciones

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|-----------|
| Lead genera un plan malformado | Media | Alto | Parser robusto con fallback a `default_phases()` |
| Plan con ciclos de dependencia | Baja | Alto | Validación de DAG antes de crear tareas |
| Frontend lento con muchas agent lanes | Baja | Medio | Virtualización de lanes, colapso automático de completadas |
| Extended thinking no disponible en todos los providers | Alta | Bajo | `chunk_type: "thinking"` solo se emite cuando el adapter lo soporta; resto sigue con solo output |
| Latencia de lead_intake aumenta (genera plan) | Media | Bajo | El Lead ya hace analysis; agregar el bloque WORKFLOW_PLAN es ~100 tokens extra |
| Tests existentes se rompen con flujo de dos pasos | Alta | Alto | Mantener `default_phases()` idéntico al flujo actual; test suite no debería cambiar |

## Criterios de aceptación

### Feature A: Visibilidad
- [ ] El usuario ve un panel con una lane por agente activo
- [ ] Cada lane muestra: rol, nombre de fase, estado (thinking/outputting/done), texto en vivo
- [ ] Las lanes completadas muestran un preview del output (colapsable)
- [ ] Los "thinking chunks" se muestran en sección separada y colapsable
- [ ] La barra de progreso existente sigue funcionando (backward compat)
- [ ] Tests existentes pasan sin modificación (backward compat total)

### Feature B: Workflow dinámico
- [ ] El Lead produce un `[WORKFLOW_PLAN]` válido para al menos 3 tipos de pedidos distintos
- [ ] El plan se parsea correctamente y las WorkTasks se crean dinámicamente
- [ ] Si el Lead no produce un plan válido, el sistema hace fallback a `default_phases()`
- [ ] El frontend muestra el estado "planificando..." mientras lead_intake corre
- [ ] Las fases aparecen dinámicamente en el UI después de que lead completa
- [ ] Tests existentes pasan (la suite de 346 tests no debe regresar)
