## Auditoría de requests

- **0 fallos** (todo 200 OK), sin CORS ni timeouts.
- Ciclo de polling: **~17 GETs por iteración** — issues, agents, runs(100),
  timeline(300, 6 UNIONs), comments+interactions **por cada issue** (fan-out
  lineal: 20 issues = 40+ requests extra), plan doc, chat(120), workspace/files,
  projects, budget, costs/summary, loop-health.
- Intervalos: 20 s en reposo, **2 s con run activo** → ~510 requests/min
  durante ejecución. SQLite local lo aguanta, pero es la principal deuda de
  eficiencia y la que peor escala con el tamaño del proyecto.

## Frentes restantes, por orden recomendado

### 1 · Consolidar el polling (deuda confirmada por la auditoría) — esfuerzo medio

Sin llegar a SSE todavía, un paso intermedio barato:
- Endpoint agregado `GET /api/project/state?since=<cursor>` que devuelva en UNA
  respuesta lo que hoy son 17: issues+threads con cambios desde el cursor,
  runs recientes, chat delta, loop-health y costes.
- El fan-out por issue desaparece (el backend hace el join una vez).
- Mantener el intervalo 2 s/20 s; el ancho de banda y las queries caen ~90%.
- SSE/WebSocket queda como fase 2 cuando haya multi-proyecto.

### 2 · B: criterios de aceptación estructurados + test_runner en el gate — esfuerzo alto, el más valioso

- `create_issue` acepta `acceptance_criteria: []`; ya existe `evidence_required`
  en el payload de delegación — unificar.
- El AGENT-REPORT del reviewer se extiende con `criteria_coverage` (criterio →
  met/unmet/evidencia). `_machine_close_verification` muestra la cobertura.
- Si el workspace contiene tests, el cierre exige report de `test_runner` con
  exit code 0 (el rol ya existe, falta hacerlo obligatorio en `_all_children_done`).

### 3 · D: política de autonomía para interacciones — esfuerzo medio ✅ (2026-07-09)

- Campo `autonomy` por proyecto (`supervised` | `autonomous`) en project_config.
- En autonomous: interacciones operativas (reintentos, reasignaciones, fix
  cycles) se auto-resuelven con default seguro; solo llegan al usuario las de
  producto (`initial_cycle_ready`, alcance).
- Expiración con default: estado `expired` ya existe en el schema; un timer en
  el heartbeat resuelve interacciones operativas sin respuesta en N min.
- **Implementado**: clasificación en `aiteam/policies.py`
  (`OPERATIONAL_INTERACTION_DEFAULTS`), reconciler `aiteam/autonomy.py` en el
  heartbeat (una vez por issue+reason; la repetición se promociona al usuario),
  TTL supervisado opt-in vía `AITEAM_INTERACTION_TTL_MINUTES`, toggle en la
  pestaña Config (`POST /api/project/autonomy`), audit `interaction.auto_resolved`.
  Nota: la resolución usa el vocabulario existente (accepted), no el estado
  `expired`; el trigger queda en `result_json.resolution_data.autonomy_trigger`.

### 4 · E/A5: cost circuit breaker — esfuerzo bajo (la base A2 ya está)

- Regla en el heartbeat: si un subtree acumula > X¢ (`actual_cost_cents`) sin
  nuevos workspace changes desde hace M runs → pausar subtree + interacción con
  el desglose de gasto. Con datos de A2 es una query.

### 5 · G: dedupe de wakeups por (agente, issue) — esfuerzo bajo

- Invariante en `enqueue_wakeup`: si ya existe un wakeup `queued` para el mismo
  (agent_id, issue_id), coalescer en vez de crear otro (hoy la idempotencia es
  por clave textual y cada fuente usa la suya).

### 6 · Pulido de UI detectado en la auditoría — esfuerzo trivial

- Punto rojo/ámbar para "Run fallida" en la timeline (hoy sale verde).
- Corregir el path de `adapter_profiles.json` mostrado en Equipo (Windows).
- Mostrar `policy_deviations` de loop-health como aviso en la vista Equipo
  (los 4 workers de pago ya se detectan; falta pintarlos).

### 7 · H: paralelismo por canal + sesiones persistentes — esfuerzo alto, posponer

- Ejecutar en paralelo runs de canales distintos (local vs API) usando el
  governor como árbitro. Toca el corazón del heartbeat: hacerlo tras 1-5.
- Sesiones persistentes de CLI (columnas `session_id_before/after` ya existen,
  siempre null): retomar sesión de codex ahorra re-derivar contexto.

## Orden sugerido de ejecución

6 (trivial, hoy) → 5 (bajo) → 4 (bajo) → 1 (medio) → 3 (medio) → 2 (alto) → 7 (alto).
El 2 es el de más valor de producto; se pone después de 1/3/4/5 solo porque
esos reducen ruido y coste de iterar mientras se desarrolla el 2.
