## EPIC-8: Lead Control Protocol (LCP) — Flujo adaptativo completo

> El Team Lead (modelo avanzado) puede tomar **cualquier decisión de flujo en tiempo real**
> emitiendo directivas estructuradas en su output. El orquestador las detecta y actúa sin
> necesitar que el usuario intervenga. El Lead es el único árbitro del flujo mid-run.
>
> Objetivo: eliminar casos en los que el Lead "sabe" la respuesta óptima pero el sistema
> lo fuerza a ejecutar un flujo rígido (scout → intake → fases → close).

### Arquitectura: directivas del Lead

El Lead emite directivas al final de su output con el formato `[DIRECTIVE: payload]`.
El orquestador las detecta al completar `lead_intake` y actúa antes de PASO 2.
Múltiples directivas pueden coexistir; se procesan en orden de prioridad.

**Tabla de directivas (orden de prioridad descendente):**

| Directiva | Payload | Cuándo usarla | Efecto en el flujo |
|-----------|---------|--------------|-------------------|
| `[REJECT]` | `"razón"` | Petición fuera de scope o imposible | Termina el run con explicación; no lanza agentes |
| `[ABORT_PHASES]` | `"razón"` | Respuesta ya clara en los briefings | Lead cierra directamente, sin fases intermedias |
| `[CLARIFY]` | `"pregunta"` | Ambigüedad bloquea el plan | Pausa el run; usuario responde; run continúa (**ya implementado**) |
| `[DIRECT_ANSWER]` | *(ninguno)* | Consulta de contexto/orientación sin build | Lead responde inline; no se spawna ningún agente |
| `[RUN_MODE]` | `planning_only` / `team_decision` | El Lead quiere una corrida deliberativa sin build | Usa presets no ejecutores; `WORKFLOW_PLAN` explícito sigue teniendo prioridad |
| `[ESCALATE]` | `complexity=high criticality=critical` | Tarea más compleja/crítica de lo estimado | Sube complexity/criticality para gates del resto del run |
| `[REPLAN]` | bloque `WORKFLOW_PLAN` nuevo | El plan inicial no es el óptimo | Reemplaza las fases dinámicas con el nuevo plan |
| `[SKIP]` | `"phase_id1 phase_id2"` | Fases innecesarias para este caso | Cancela esas fases del plan activo |
| `[ADD_PHASE]` | `role "objetivo"` | Falta una fase no prevista | Inserta nueva fase en el plan |
| `[EXTEND_BUDGET]` | `+N` | Tarea más larga de lo estimado | Suma N rondas al round_budget del run |
| `[SET_BUDGET]` | `N` | El Lead quiere recortar o ampliar el run a un presupuesto concreto | Fija el `round_budget` absoluto del run, también mid-run |
| `[FORCE_GATE]` | `"phase_id"` | QA detecta bug crítico que hay que regate | Añade gate adicional a una fase ya completada |
| `[RETRY_ROUTE]` | `"phase_id"` | El Lead quiere reintentar una fase con otra ruta/modelo | Reabre esa fase y excluye el último adapter usado antes de reintentar |
| `[DELEGATE]` | `scout_type "query"` | Necesita datos específicos mid-task | Lanza scout on-demand y espera resultado (**E7-C**) |

### Casos de uso concretos

```
"¿de qué iba el proyecto?" → scouts briefing suficiente → [DIRECT_ANSWER] → responde sin agentes
"refactoriza el módulo de auth" → Lead lee scout → [ESCALATE: complexity=high] → gates estrictos
"añade un endpoint CRUD" → simple → [SKIP: "review"] → solo build + qa
"implementa pagos con Stripe" → Lead no sabe qué versión → [CLARIFY: "¿Stripe v2 o v3?"]
"haz algo con los datos" → imposible sin más info → [REJECT: "Necesito saber qué datos y qué transformación"]
"optimiza la DB" → Lead ve que también hay índices → [ADD_PHASE: engineer "Crear índices faltantes"]
"¿qué errores hay en el proyecto?" → [ABORT_PHASES] + respuesta directa del Lead con lo que saben los scouts
run de 15 rondas necesita 20 → [EXTEND_BUDGET: +8]
"el build salió, pero quiero otra opinión de modelo" → [RETRY_ROUTE: "build"]
"ya hay suficiente información, recorta el run a 3 rondas" → [SET_BUDGET: 3]
```

### Invariantes de diseño LCP

1. **El Lead nunca bloquea el sistema** — si emite una directiva malformada, se ignora y el flujo continúa normalmente
2. **Una sola directiva por categoría** — si hay dos `[ESCALATE]`, se usa la última
3. **`[CLARIFY]` tiene precedencia sobre todas excepto `[REJECT]`** — no tiene sentido planificar si hay ambigüedad bloqueante
4. **Las directivas se eliminan del output antes de mostrarlo al usuario** — el usuario ve solo el texto limpio del Lead
5. **El Lead puede combinar directivas** — `[ESCALATE: complexity=high] [SKIP: "review"]` es válido
6. **Trazabilidad** — cada directiva procesada emite un evento en `events.jsonl` con tipo `lcp_directive_applied`

## EPIC-11: Context Curator — compactacion economica de memoria por proyecto

> **Motivacion (2026-03-31):**
> OpenAI/Codex documenta que los subagentes ayudan a evitar `context pollution`
> y `context rot` moviendo trabajo ruidoso fuera del hilo principal y devolviendo
> **summaries en lugar de output crudo**. El proyecto ya compacta algo de memoria
> en `aiteam/memory.py` y `aiteam/agent_session.py`, pero todavia no existe una
> capa barata y semantica de compactacion por proyecto/chat root.

### Gap actual

- `AgentMemoryStore` solo compacta por numero de entradas; no destila semantica.
- `agent_session.py` resume turnos viejos, pero a nivel de thread local.
- `project_continuity` agrega contexto reciente, pero no mantiene una memoria por
  capas con invalidez, procedencia y decisiones vigentes.
- El Lead todavia puede cargar demasiado contexto historico si una run larga se
  reanuda muchas veces o si hay muchos `delegate_batches` / `specialist_reports`.

### Objetivo

Crear un especialista barato `context_curator` que mantenga una memoria viva por
proyecto y por `chat_root`, reduciendo el contexto bruto que llega al Lead sin
perder hechos durables, decisiones vigentes ni preguntas abiertas.
