# DESIGN 2026-05-12 · Lead-as-Evaluator, QA Absorption, Tier Discipline, Block Curator

> Objetivo: refactorizar el framework para que el Lead sea el **único evaluador** que decide quién hace cada acción según (criticidad, complejidad), eliminar el rol QA como entidad separada, endurecer la frontera Tier 2 ↔ Tier 3, y rediseñar el context curator para que trabaje en **bloques incrementales** tanto para el Lead como para el chat del usuario.

---

## 1 · Razonamiento de fondo

### 1.1 Por qué el QA debe ser función del Lead, no opcional

**Estado actual:**
- `qa` es un rol Tier 2 con adapter, agente, runs y costes propios.
- El skill `reviewer.md` ya admite que absorbe el trabajo de QA estático: *"Static QA (you own this — there is no separate QA agent by default)"*.
- `_all_children_done` tiene un gate de calidad para reviewer y (desde hoy) para QA, pero ambos se solapan.
- La decisión "¿esto está terminado?" la toma actualmente `_all_children_done` (gate booleano) + la interacción `initial_cycle_ready` que el usuario confirma. No hay un evaluador agentivo intermedio.

**Tensión:** un rol QA que sólo hace "lectura de archivos + análisis estático" es exactamente lo que ya hace el reviewer. Su única razón de existir distinta sería **ejecución de comandos reales** (correr `pytest`, abrir un browser headless, etc.), pero los agentes API-only no pueden hacerlo.

**Decisión arquitectónica:**
- **El gate de calidad lo decide el Lead** consumiendo: workspace evidence + reviewer report + (opcional) test_runner report.
- Se elimina el rol `qa` como Tier 2. Su capacidad "ejecutar tests reales" se descompone en un nuevo **Tier 3 `test_runner`** (mecánico, sólo ejecuta y reporta).
- "No opcional" significa: el Lead **siempre** evalúa antes de cerrar. No hay flag para saltarse el gate.

### 1.2 Por qué el reviewer NO debe pisar el territorio del scout

**Síntomas observados:**
- Reviewers que listan árboles de archivos enteros "para referencia" (eso es trabajo de file_scout).
- Reviewers que buscan información en la web (web_scout).
- Reviewers que resumen el hilo del proyecto (context_curator).

**Causa raíz:** sin reglas firmes, un agente Tier 2 con tokens disponibles tiende a "ser exhaustivo" y absorbe trabajo de otros.

**Principio rector:**
- **Tier 2 = juicio sobre artefactos.** Lee un input acotado (workspace_files, reporte del engineer, criterios) y emite un veredicto.
- **Tier 3 = recolección de datos.** Lee fuentes y devuelve resumen plano, sin veredicto.

**Consecuencia:** si el reviewer necesita contexto adicional (un archivo no presente, un fact externo), debe emitir `result: blocked` con `blocker: needs_scouting_for_<topic>` y dejar al Lead orquestar el scout.

### 1.3 El núcleo: Lead-como-evaluador

El usuario lo formuló así: *"cada accion (sea programar, scoutear, leer documentos...) debe ser valorada por el lead, y valorar su criticalidad y su dificultad"*.

Esto es un cambio arquitectónico mayor: el Lead deja de ser un **dispatcher** ciego (que crea engineer+reviewer+qa por defecto) y se convierte en un **evaluador-router**.

**Por cada acción identificada, el Lead asigna dos puntuaciones:**

| Eje          | Valores                  | Definición                                                                        |
|--------------|--------------------------|-----------------------------------------------------------------------------------|
| Criticidad   | low / medium / high / critical | Cuánto cuesta equivocarse. Crítico = bloquea entrega, alto = afecta calidad final. |
| Complejidad  | low / medium / high      | Cuánto razonamiento/contexto requiere. Alto = requiere visión arquitectónica.    |

**Tabla de enrutamiento (combinación cardinal):**

```
                          criticidad
                  low      medium    high     critical
complejidad
   low      →    Tier 3   Tier 3   Tier 2   Lead-self
   medium   →    Tier 3   Tier 2   Tier 2   Lead-self
   high     →    Tier 2   Tier 2   Lead-self Lead-self
```

Justificación de cada celda:
- **(low, low)**: trivial — file_scout / test_runner basta.
- **(low, high)**: criticidad baja pero requiere visión → Tier 2 senior puede hacerlo.
- **(critical, low)**: cualquier paso crítico, por simple que parezca, lo hace el Lead (porque equivocarse aquí cuesta).
- **(high, high) / (critical, *)**: Lead-self obligatorio.

**Implementación:** función pura `route_action(criticality, complexity, action_type) -> Routing`. Tres valores de retorno: `TIER_3`, `TIER_2`, `LEAD_SELF`. El `action_type` (`code`, `review`, `scout`, `research`, `synthesis`, `test_exec`) refina la elección del rol concreto dentro de la tier.

### 1.4 Lead-self: cómo el Lead "hace" trabajo

El Lead actual es `lead_builtin` — una clase Python sin LLM propio para ejecutar tareas. Si el Lead debe "hacer trabajo crítico él mismo", necesita un brazo ejecutor.

**Decisión:** crear un nuevo rol `lead_executor` (Tier 1, senior).

- `lead_executor` no es un agente distinto del Lead — es la **identidad de ejecución del Lead**.
- Adapter: senior LLM (mismo que use el Lead para generar planes; el usuario configura cuál).
- Cuando `route_action` devuelve `LEAD_SELF`, el Lead crea un hijo `role=lead_executor` con la acción concreta, asignado a un agente senior.
- El `lead_executor` reporta vía `notify_supervisor` al `lead_builtin`, igual que cualquier Tier 2.

Esto preserva la separación entre **decisión** (lead_builtin, gratis) y **ejecución** (lead_executor, costoso) y permite contabilizar correctamente los tokens senior.

### 1.5 Block-based context curator

**Estado actual:** se dispara una vez cuando `comments ≥ 8 AND no plan doc`. Una vez done, bloquea futuros spawns para evitar loops infinitos (fix de hoy).

**Lo que pide el usuario:**
- Disparo por tamaño de contexto (tokens/caracteres), no por número de comentarios.
- Síntesis **incremental por bloques**: cada disparo añade un bloque nuevo cubriendo el rango de comentarios desde la última síntesis.
- Lo mismo aplicado al chat del usuario en la UI, reduciendo a ≤20% el espacio visible y manteniendo el chat completo accesible vía popup.

**Modelo de almacenamiento:**

Documento `issue_documents.key='context_summary'` con cuerpo JSON estructurado:

```json
{
  "blocks": [
    {
      "block_index": 1,
      "comment_range": {"start_id": "c-001", "end_id": "c-015"},
      "comment_count": 15,
      "char_count_original": 12450,
      "char_count_summary": 2200,
      "compression_ratio": 0.177,
      "summary_markdown": "## Fase 1 (decisiones iniciales)\n- ...",
      "created_at": "2026-05-12T14:00:00Z",
      "synthesized_by_run_id": "run-abc123"
    },
    {
      "block_index": 2,
      "comment_range": {"start_id": "c-016", "end_id": "c-030"},
      ...
    }
  ],
  "synthesized_through_comment_id": "c-030",
  "last_run_id": "run-def456"
}
```

**Trigger:** `_maybe_spawn_context_curator` se reescribe para:
1. Calcular `unsynthesized_char_count` = suma de `length(body)` de comentarios con `rowid > rowid(synthesized_through_comment_id)` (o todos si no hay síntesis previa).
2. Si `unsynthesized_char_count >= _CONTEXT_CURATOR_CHAR_THRESHOLD` (por defecto 8000 chars ≈ 2000 tokens) → spawnear curator.
3. El curator escribe un **nuevo bloque** (no reescribe los existentes).
4. La idempotencia se respeta: no spawnear si hay un curator activo (todo/in_progress/blocked).

**Wake payload del Lead/Tier-2 actualizado:**

```json
{
  "issue": {...},
  "context_summary": {
    "blocks": [...],           // bloques resumidos verbatim
    "blocks_total_chars": 8800,
    "synthesized_through": "c-030"
  },
  "comments_recent": [...],    // comentarios después de synthesized_through, verbatim
  "comments_total": 47,
  "compression_active": true
}
```

El Lead ve: contexto antiguo comprimido + comentarios recientes verbatim → cabe en su ventana sin pagar el coste total del hilo.

### 1.6 User chat compression

**Estado actual:** el thread visible al usuario es la lista cruda de `issue_comments`. La UI no comprime.

**Diseño:**

Backend:
- Endpoint nuevo: `GET /api/issues/{id}/thread?view=compact|full`
- `view=compact` (default): devuelve `{summary_blocks, recent_comments, total_comments}` — usa el mismo `context_summary` que consume el Lead.
- `view=full`: devuelve la lista cruda de `issue_comments` ordenada cronológicamente.
- Garantía: el endpoint nunca pierde data — `full` siempre disponible.

Frontend:
- Componente `ThreadView` con dos modos:
  - Compact (default): renderiza bloques resumidos colapsables + comentarios recientes verbatim.
  - Full: modal/popup que carga `view=full` y renderiza el hilo completo cronológico.
- Botón "Ver hilo completo" abre el modal.
- En compact, cada bloque resumido muestra: `comment_range`, `comment_count`, `compression_ratio`, `summary_markdown`.

**Target de compresión:** el skill del curator debe producir resúmenes ≤ **20% del tamaño original** del bloque (medido en chars). Si no llega, indicar qué se omitió y por qué.

### 1.7 Tier 2 — disciplina estricta de scope

| Rol             | Tier | Acepta como input             | Produce                              | NO debe hacer                                                   |
|-----------------|------|--------------------------------|---------------------------------------|----------------------------------------------------------------|
| `engineer`      | 2    | spec + workspace_files         | file_ops + AGENT-REPORT(done\|partial\|blocked) | Pedir archivos al Lead; explorar más allá del scope; revisar su propio código |
| `reviewer`      | 2    | workspace_files + engineer report | comment + AGENT-REPORT(done\|changes_requested\|blocked) | Scoutear archivos no provistos; investigar web; resumir threads |
| `lead_executor` | 1    | spec + cualquier contexto del Lead | artefactos (code/review/research) según action_type | (sin restricciones — es la mano del Lead)                       |
| `file_scout`    | 3    | lista de paths                 | comment con extracción                | Emitir veredictos; sugerir cambios                              |
| `web_scout`     | 3    | query                          | comment con resumen de búsqueda       | Decidir arquitectura; opinar sobre código                       |
| `context_curator` | 3  | target_issue_id + threshold    | bloque nuevo en `context_summary`     | Crear sub-issues; interacciones; opinar                          |
| `test_runner`   | 3    | comando(s) + cwd               | comment con stdout/exitcode           | Modificar archivos; sugerir fixes                                |

Reglas duras codificadas en cada skill + validadas en `work_contract.py`:
- Tier 3 → permite sólo `add_comment` + `set_status: done`. Prohíbe `create_issue`, `create_interaction`, `update_plan`, `write_file*`.
- Reviewer (Tier 2) → no `create_issue`. Si necesita scouting, `result: blocked` con `blocker: needs_scouting_for_<topic>`.

### Fase 3 · Modelo de scoring de acciones

**Goal:** introducir `route_action(criticality, complexity, action_type) -> Routing` como función pura y obligar al plan-generator del Lead a producir scores por acción.

**Cambios en código:**
- `aiteam/action_routing.py` (nuevo):
  ```python
  from enum import Enum
  
  class Routing(str, Enum):
      TIER_3 = "tier_3"
      TIER_2 = "tier_2"
      LEAD_SELF = "lead_self"
  
  _CRIT_SCORE = {"low": 0, "medium": 2, "high": 4, "critical": 6}
  _COMP_SCORE = {"low": 0, "medium": 2, "high": 4}
  
  def route_action(
      *, criticality: str, complexity: str, action_type: str
  ) -> Routing:
      """Pure routing decision based on (criticality, complexity)."""
      score = _CRIT_SCORE.get(criticality, 2) + _COMP_SCORE.get(complexity, 2)
      # action_type can shift the boundary (e.g. test_exec is always Tier 3)
      if action_type == "test_exec":
          return Routing.TIER_3
      if score >= 8:
          return Routing.LEAD_SELF
      if score >= 4:
          return Routing.TIER_2
      return Routing.TIER_3
  
  def pick_role_for_routing(
      routing: Routing, action_type: str
  ) -> str:
      """Map (routing, action_type) → concrete role name."""
      if routing == Routing.LEAD_SELF:
          return "lead_executor"
      if routing == Routing.TIER_2:
          return {
              "code": "engineer",
              "review": "reviewer",
              "synthesis": "lead_executor",  # synthesis crítico siempre Lead
          }.get(action_type, "engineer")
      # Tier 3
      return {
          "scout_files": "file_scout",
          "scout_web": "web_scout",
          "synthesis": "context_curator",
          "test_exec": "test_runner",
      }.get(action_type, "file_scout")
  ```
- `aiteam/lead_intake.py`: cuando el LLM propone un plan, cada item debe incluir `{criticality, complexity, action_type}`. `apply_accepted_team_proposal` los pasa por `route_action` + `pick_role_for_routing` para decidir el rol concreto del hijo.
- `aiteam/adapters/work_contract.py`: el `create_issue` op schema añade `action_type` opcional (`code | review | scout_files | scout_web | research | synthesis | test_exec`). Si el agente no lo especifica, se infiere de `role`.
- `aiteam/heartbeat/executor.py`: 
  - `_create_delegated_issue` consulta `route_action` cuando recibe `criticality+complexity+action_type` en el spec, ignorando el `role` propuesto si el routing decide otro. Esto **fuerza** la disciplina incluso si el LLM se equivoca.
  - Logging: cada decisión de routing se loguea con `log_activity(action="action.routed", payload={criticality, complexity, action_type, routing, role})`.

**Tests (`tests/test_action_routing.py`, nuevo):**
- Tabla exhaustiva: cada celda de la matriz 4×3 verificada.
- `test_test_exec_always_tier3` independientemente de scores.
- `test_lead_self_returns_lead_executor_role`.
- `test_pick_role_synthesis_critical_goes_to_lead_executor`.
- `test_create_delegated_issue_overrides_role_per_routing`.
- Integración: Lead recibe plan con un item `{criticality: critical, complexity: high, action_type: code}` → crea child con `role=lead_executor`, no `engineer`.

**Criterios de aceptación:**
- Pura: `route_action` es deterministica y testeable sin DB.
- Cobertura de matriz 100%.
- El Lead no puede crear un engineer para una tarea critical+high (el routing lo overridea).

### Fase 5 · Block-based context curator

**Goal:** reescribir el curator y su trigger para trabajar en bloques incrementales basados en tamaño de contexto.

**Cambios en código:**
- `aiteam/heartbeat/executor.py`:
  - Reemplazar `_CONTEXT_CURATOR_COMMENT_THRESHOLD = 8` por `_CONTEXT_CURATOR_CHAR_THRESHOLD = 8000`.
  - `_maybe_spawn_context_curator` reescrito:
    1. Leer `context_summary` doc (si existe), extraer `synthesized_through_comment_id`.
    2. Calcular `unsynthesized_char_count` = `SUM(LENGTH(body)) FROM issue_comments WHERE issue_id=? AND (synthesized_through IS NULL OR rowid > rowid_of_synthesized_through)`.
    3. Si `unsynthesized_char_count < threshold` → return.
    4. Idempotencia: no spawnear si hay curator no-cancelado ACTIVO (status in todo/in_progress/blocked). Un curator `done` NO bloquea más (porque ya añadió su bloque; el siguiente cubre los nuevos comentarios).
    5. Crear curator child con description que incluya: `Target issue: <id>`, `Synthesize block starting from comment: <id>` (o "all" si primer bloque).
- `aiteam/db/documents.py`: añadir helper `append_summary_block(issue_id, block_dict)` que:
  - Carga el doc actual `key='context_summary'` (o crea uno con `blocks: []` si no existe).
  - Añade el bloque al array `blocks`.
  - Actualiza `synthesized_through_comment_id` = `block.comment_range.end_id`.
  - Persiste con `put_document`.
- `aiteam/db/wake_payload.py`:
  - Si `context_summary` doc existe, leerlo y exponer en payload como `context_summary.blocks` + `context_summary.synthesized_through`.
  - `comments` del payload se filtran a "después de synthesized_through" (más recientes).
- `skills/context_curator.md`: reescribir para describir el modelo de bloques:
  - Recibe `Target issue: <id>` y `Synthesize from: <comment_id|all>`.
  - Lee comentarios desde ese punto en adelante.
  - Produce un resumen markdown ≤ **20% del tamaño original** del rango.
  - Llama POST/PUT endpoint nuevo `POST /api/issues/{target_id}/context-summary/blocks` con `{summary_markdown, comment_range, char_count_original}` (la API hace `append_summary_block` server-side).
  - Cierra su task issue con `result: done`.

**API nueva:**
- `POST /api/issues/{id}/context-summary/blocks`: body `{summary_markdown, start_comment_id, end_comment_id, char_count_original}`. Persiste el bloque.
- `GET /api/issues/{id}/context-summary`: devuelve el doc completo.

**Tests (`tests/test_block_curator.py`, nuevo):**
- `test_no_spawn_when_unsynthesized_chars_below_threshold`
- `test_spawn_when_unsynthesized_chars_above_threshold_first_time`
- `test_second_block_spawned_when_new_comments_exceed_threshold`
- `test_no_spawn_when_active_curator_exists`
- `test_done_curator_does_not_block_when_new_chars_accumulate`
- `test_append_summary_block_idempotent_on_same_range` (si end_id ya está en `synthesized_through`, no-op)
- `test_wake_payload_uses_summary_blocks_when_doc_exists`
- `test_compression_ratio_enforced_in_summary` (skill behavior — tested via runtime assertion on block char_count_summary / char_count_original ≤ 0.20 + 0.05 margen)

**Migración:**
- Issues con `plan` doc existente NO se ven afectadas (el plan doc tiene key distinto).
- El campo `_CONTEXT_CURATOR_COMMENT_THRESHOLD` permanece como deprecated alias para no romper tests existentes; emite `DeprecationWarning`.
- `tests/test_context_curator_auto_trigger.py` se REESCRIBE entero para el nuevo modelo (no se mantienen los tests basados en comment_count).

**Criterios de aceptación:**
- Hilo de 20K chars genera 2-3 bloques separados, no uno solo.
- Cada bloque cubre un rango disjoint de comentarios.
- El Lead recibe en su payload los bloques + los comentarios "frescos" sin duplicación.

---

### Fase 6 · API + UI de compresión del chat de usuario

**Goal:** exponer el `context_summary` a la UI con vista compact (default) y full (popup).

**Cambios backend (`api/routers/issues.py`):**
- `GET /api/issues/{id}/thread?view=compact|full`:
  - `view=full`: alias de `GET /api/issues/{id}/comments` ordenado cronológicamente, sin agregación.
  - `view=compact` (default): devuelve `{summary_blocks, recent_comments, total_comments, synthesized_through, has_compression}`.
  - `recent_comments`: comentarios con `rowid > rowid(synthesized_through)`.
  - Si no hay `context_summary` doc: `summary_blocks: []`, `has_compression: false`, `recent_comments` = todos los comentarios.
- Tests `api/tests/test_thread_router.py`: ambos modos, casos sin doc, con doc, con doc parcial.

**Cambios frontend (asumiendo React/TS típico en `web/`):**
- Componente nuevo `<ThreadView issueId={id} />`:
  - Estado: `mode: 'compact' | 'full'`.
  - Fetch inicial: `view=compact`.
  - Render compact:
    - Cada `summary_block` como `<Card>` colapsable con header "Bloque N (comments c-X..c-Y, ratio R%)" y body `summary_markdown`.
    - Luego `<CommentList comments={recent_comments} />`.
    - Footer con botón "Ver hilo completo".
  - Botón → `<Modal>` con fetch `view=full` y render lineal cronológico.
- Estilo: bloques colapsables por default, primer bloque expandido si sólo hay 1.
- Tests E2E (Cypress/Playwright si existen, si no smoke tests vía storybook): 
  - Renderiza N bloques.
  - Botón abre modal con full thread.
  - Sin compresión, renderiza directamente como hoy.

**Criterios de aceptación:**
- Hilo de 50 comentarios con 3 bloques de síntesis se muestra como 3 bloques colapsables + ~10 comentarios recientes.
- El popup carga el hilo completo en ≤ 1s para hilos ≤ 200 comentarios.
- Sin sumario disponible, la vista funciona idéntica a la actual (no regresión).

## 3 · Riesgos y mitigaciones

| Riesgo                                                                                    | Probabilidad | Impacto | Mitigación                                                                                                       |
|-------------------------------------------------------------------------------------------|--------------|---------|------------------------------------------------------------------------------------------------------------------|
| El LLM no devuelve `criticality+complexity+action_type` en su plan                         | Media        | Alto    | Fase 3 define defaults conservadores (medium/medium/code) y loguea WARNING; suite tiene fixture de fallback.    |
| `lead_executor` se vuelve caro y se usa por defecto                                       | Media        | Alto    | Tests verifican que ninguna combinación con criticidad ≤ high y complejidad ≤ medium produce `LEAD_SELF`.       |
| Eliminar QA rompe proyectos existentes con agentes QA en producción                       | Baja         | Medio   | Migración no destructiva: agentes QA existentes siguen funcionando, sólo no se crean nuevos por defecto.       |
| El curator produce bloques pobres (no respeta ratio ≤ 20%)                                 | Media        | Bajo    | Skill incluye el ratio como contrato explícito; la API valida `char_count_summary / char_count_original ≤ 0.30` (margen) y rechaza con error si excede. |
| Block-based curator infinito si el ratio falla repetidamente y los comentarios siguen llegando | Baja      | Medio   | Hard cap: máximo N bloques por issue (e.g. 10). Después, sólo se acepta `view=full` y no se sintetiza más.       |
| El reviewer empieza a abusar de `blocker: needs_scouting_for_X`                            | Media        | Bajo    | Telemetría: contar cuántos reviews por issue terminan blocked; alertar si excede 2.                              |
| UI compact view confunde al usuario que esperaba ver todo el hilo                          | Media        | Bajo    | Botón "Ver hilo completo" prominente; toggle persistente en localStorage para usuarios que prefieran full siempre. |

## 7 · Decisiones que requieren confirmación del usuario antes de empezar

Sonnet debería **detenerse y preguntar al usuario** antes de la Fase 3 sobre estos tres puntos:

1. **¿`lead_executor` usa el mismo adapter que `lead_builtin` para plan generation, o uno configurado por separado?**
   - Default propuesto: mismo adapter. Más simple. El usuario configura un único "senior LLM" para Lead.

2. **¿La matriz de routing (sección 1.3) es la correcta o el usuario prefiere ajustar umbrales?**
   - Default propuesto: tal como está. Critical+low → Lead-self refleja "lo importante lo hace el más capaz aunque sea fácil".

3. **¿El compression ratio target de ≤20% es estricto (la API rechaza si excede) o aspiracional (sólo log)?**
   - Default propuesto: la API valida ≤30% (20% objetivo + 10% margen) y rechaza con error si excede; el skill apunta a 20%.

Si el usuario aprueba los defaults sin cambios, Sonnet procede.
